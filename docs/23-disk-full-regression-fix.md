# 23. 実環境スコア低下（485k→371k）の原因と修正：run中のディスク満杯

## 症状
実環境で **fail 6701**（≒ -134k ペナルティ）。失敗は GET / だけでなく **/@user 多数・css/js/favicon・画像・POST 系まで全エンドポイント**。
スコアが 485,121 → 371,233 に低下（success 自体はむしろ増えていた）。

## 原因：実行中にアップロード画像でディスクが満杯になる
- アップロード画像は run 中ディスクに書かれ、次の `/initialize` まで消えない＝**run 中ずっと増え続ける**。
- 計測：90s ベンチで **ディスク空きが約 1GB 減**（高スループットほど速い）。
- ローカルで空きを ~700MB に絞って再現 → run 中に 0MB になり、`POST /→500`・画像 404 が多発（fail 794）。
- 実環境は、ローカルで行ったディスク掃除（**孤立 binlog 2.5GB 削除・ログ truncate は git 管理外**）が反映されておらず空きが少ない。さらに今回の最適化で**スループットが上がりアップロード量が増加** → run 中に満杯。
- 満杯になると MySQL 書き込み・画像書き込みが失敗。書き込みが詰まると nginx の upstream 接続が滞留して worker 接続を食い尽くし、**読み取り・静的配信まで巻き込んで全滅**（"リクエストに失敗しました"）。

## 修正

### 1. ヘッドルーム確保（根本）: `bin/free-disk.sh`（再現可能・実環境で実行）
- 孤立 binlog 削除（binlog 無効時）／binlog が有効なら警告＋PURGE。
- slow log・access log・journal の truncate。
- 残留アップロード画像（id>10000）の削除。
- **実環境で公式計測前に必ず実行**（binlog は `bin/apply-infra.sh` で無効化されている前提）。

### 2. 連鎖を防ぐ防御（コード）
- **panic リカバリ**: `chi middleware.Recoverer` を追加。1 ハンドラの panic でプロセス全体が落ちて全リクエストが失敗する事故を防ぐ。
- **書き込み失敗を可視化しない**: `writeImageFile` がエラーを返すように変更し、画像書き込みに失敗した投稿は **キャッシュに載せない**（壊れた画像で他リクエストの一覧描画まで失敗させない）。POST は 500 で完結。
- 読み取りは完全キャッシュ化済みなので、**ディスクが満杯でも GET 系・静的配信は生存**するようになった。

## 検証
- 通常 60s: 337,201 / fail 0（リグレッションなし）。
- 空き ~700MB に絞った run 中（ディスク 220MB まで低下）でも **GET / と GET /@mary が 200・isu-go 生存**。以前は全滅していた。

## 実環境での手順（重要）
```bash
# 最新を deploy 後、アプリサーバで：
bin/apply-infra.sh     # MySQL binlog 無効などインフラ設定を反映（未適用なら）
bin/free-disk.sh       # ディスクのヘッドルームを回復
bin/scoring-mode.sh    # 計測オーバーヘッドOFF
# その後ベンチ実行
```
`df -h /` で、**1 回の run で増える分（高スループットで ~1GB/90s）より十分大きい空き**があることを確認すること。
