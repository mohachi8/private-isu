# 05. 画像の静的配信化（nginx で /image/* を返す）— Phase 2

## 何を / なぜ

改善前は、画像は `posts.imgdata`（MySQL の BLOB）に保存され、**毎回 Go アプリが DB から読み出して返して**いました。
alp 計測では `/image/*` の応答時間合計が **約 154 秒**（全エンドポイント中2位）に達し、アプリと DB の大きな負荷源でした。

画像は一度作られたら中身が変わらない**静的コンテンツ**です。これは「**静的ファイルは nginx が直接返す**」という ISUCON の定番改善が効く典型例です。

## どうやって

**自己修復する遅延マテリアライズ方式**を採用しました。

1. **nginx**（`infra/nginx/sites-available/isucon.conf`）: `/image/` はまずディスクのファイルを探し、無ければアプリへフォールバック。
   ```nginx
   location /image/ {
     try_files $uri @app;   # ファイルがあれば nginx が直接返す
     expires 1d;
   }
   location @app { proxy_pass http://localhost:8080; }
   ```
2. **アプリ**（`webapp/golang/app.go`）:
   - `getImage`: DB から読んだ画像を `../public/image/<id>.<ext>` に**書き出してから**返す → 次回以降は nginx が配信。
   - `postIndex`（画像投稿）: アップロード画像を DB と同時にディスクへも書き出す。
   - `getInitialize`: ベンチの初期化に合わせ、投稿分（post id > 10000）の画像ファイルを削除（`cleanupUploadedImages`）。シード画像（id ≤ 10000）は不変なので残し、マテリアライズ済みを再利用。

この方式の利点: 事前の一括エクスポート不要。最初のアクセスで自動的にディスクへ載り、2回目から nginx が返す。`id` は自動採番で再利用されないため、ディスク上のファイルと内容が食い違う事故が起きない。

## 結果

| 時点 | スコア |
|---|---|
| Phase 1 後（インデックス）| 16,447 |
| Phase 2 直後（初回・マテリアライズ中）| 17,527 |
| Phase 2 ウォーム後 | **25,912** |

- `/image/*` の応答時間合計: **約 154 秒 → 約 13 秒**。配信回数 15,653 回を nginx が平均 **0.001 秒**で処理。
- アクセスログで2回目以降の画像リクエストは `apptime:-`（アプリ未経由＝nginx が直接配信）になっていることを確認。

> 補足: 初回ベンチはシード画像をディスクへ書き出すコストが乗るためスコアが伸びにくい。2回目以降がウォーム状態の実力値。

## 次の最大ボトルネック

Phase 2 後、alp の首位は **`GET /`（合計約 340 秒, 平均 0.42 秒）** と **`GET /posts`（約 110 秒）**。
どちらも `makePosts` を通り、(1) 投稿ごとのコメント／ユーザー取得の **N+1**、(2) `GET /` の `SELECT posts` が **LIMIT なしで全件取得** という2点が原因。次フェーズ（Phase 3/4）で対処する。
